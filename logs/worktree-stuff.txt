● Git submodules are indeed painful. Let me outline the alternatives:

  Alternatives to git submodules

  1. Git subtrees
  - Merges external content directly into your repo
  - Simpler day-to-day workflow (no submodule update dance)
  - Downside: loses the "optional checkout" property you want - everything is always there

  2. Git worktrees
  - Lets you check out multiple branches simultaneously in separate directories
  - Could work for your detached-branch-in-same-repo pattern
  - git worktree add misc/notes notes-branch
  - Cleaner than submodules pointing at yourself

  3. Sparse checkout (modern cone mode)
  - Clone everything but only materialize what you need
  - git sparse-checkout set to pick directories
  - Good for "minimal by default, opt-in to more"

  4. Just separate repos + a bootstrap script
  - Honestly sometimes the simplest answer
  - A ./setup.sh that clones optional deps into expected locations
  - No git magic to fight

  For your specific setup

  The "detached branches in same repo as submodules" pattern is clever but fighting git's model. Consider:

  1. Worktrees for misc/notes and misc/dist - they're branches of this repo anyway
  2. Actual separate repo for misc/mitamae (upstream sources don't belong as a branch of your config repo)
  3. Git LFS for binaries in misc/dist (as you noted)

  Want me to look at your current .gitmodules and sketch out what a worktree-based setup would look like?

  Worktrees 101

  # Add a worktree - checks out branch into a directory
  git worktree add <path> <branch>

  # List worktrees
  git worktree list

  # Remove a worktree
  git worktree remove <path>

  Key difference from submodules: worktrees share the same .git database. No separate clone, no sync issues, no "detached HEAD" confusion.

  What migration would look like

  # 1. Remove the self-referencing submodules
  git submodule deinit misc/dist misc/notes misc/mitamae
  rm -rf misc/dist misc/notes misc/mitamae

● Exactly right - worktrees are local state, not tracked in the repo. There's no .gitworktrees equivalent to .gitmodules.

  Hooks situation:
  - post-checkout exists and runs after clone (clone does a checkout at the end)
  - BUT hooks live in .git/hooks/ which isn't cloned - chicken and egg problem

  Practical options:

  1. Setup script (simplest)
  #!/bin/sh
  # setup-worktrees.sh
  git worktree add misc/dist dist 2>/dev/null || true
  git worktree add misc/notes notes 2>/dev/null || true
  git worktree add misc/mitamae mitamae-src 2>/dev/null || true
  1. People run ./setup-worktrees.sh after clone if they want extras.
  2. Makefile target (if you already have one)
  extras:
      git worktree add misc/dist dist
      ...
  3. Git config + template (fancy but fragile)
    - Set up a hook in ~/.git-templates/hooks/post-checkout
    - Only works if people have the template configured

  ---
  Honestly, the tradeoff is:

  |                      | Submodules             | Worktrees         |
  |----------------------|------------------------|-------------------|
  | Relationship tracked | Yes (.gitmodules)      | No                |
  | Works across clones  | Yes (with --recursive) | No (needs script) |
  | Mental model         | Confusing              | Simple            |
  | Day-to-day UX        | Pain                   | Nice              |

  For your case - since these are optional extras anyway - a setup script seems fine. The people who want misc/notes will run one command either way.

  Does that tradeoff work for you, or is the "declarative relationship" important enough to stick with submodules for those three?

  Branch filtering

  git filter-branch is deprecated. Use git filter-repo (much faster, less footgun-y):

  # Install if needed
  pip install git-filter-repo

  # Extract commits touching dist/mitamae-* to dist branch
  git filter-repo --path dist/mitamae-* --refs dist --force

  # Extract deploy/ to feature branch
  git checkout -b feature/deploy
  git filter-repo --path deploy/ --refs feature/deploy --force

  BUT - filter-repo rewrites history, which is destructive. If you just want to move current files to another branch without rewriting history:

  # Simpler: just move the files to the branch now, accept messy history
  git checkout dist
  git checkout master -- dist/mitamae-*
  git commit -m "Move binaries to dist branch"
  git checkout master
  git rm -r dist/mitamae-*
  git commit -m "Remove binaries from master"

  Which approach do you want? Clean history (filter-repo) or just "move it and move on"?

● filter-branch has had a big scary warning in the git docs since ~2.24 (2019):

  git filter-branch has a plethora of pitfalls... Please use an alternative history filtering tool such as git filter-repo.

  It still works, just yells at you. The main issues are: it's slow, easy to corrupt things, and the CLI is confusing.

  Hierarchy of repo surgery tools:

  | Tool          | Complexity         | Use case                                    |
  |---------------|--------------------|---------------------------------------------|
  | filter-repo   | Medium             | 90% of history rewrites                     |
  | filter-branch | Medium (but angry) | When you hate yourself                      |
  | reposurgeon   | PhD thesis         | VCS migrations, archaeology, complex grafts |

  For "move files touching X pattern to branch Y", filter-repo is the right tool. Reposurgeon is for when you're merging 3 CVS repos with a Subversion repo and need to reconstruct deleted branches from 2003.

  # Your case is probably just:

  1. --force - filter-repo is paranoid, you'll probably need this flag a lot
  2. Tags - filter-repo rewrites them too, which is usually what you want. Just be aware.

● Yeah, cherry-pick the ones you want onto a clean branch:

  # See which commits actually touch deploy/
  git log --oneline feature/deploy -- deploy/

  # Start fresh from master
  git checkout -b feature/deploy-clean master

  # Cherry-pick just those commits (in order)
  git cherry-pick $(git rev-list --reverse master..feature/deploy -- deploy/)

  The --reverse is important - rev-list outputs newest-first by default, but cherry-pick needs oldest-first.

  If that one-liner looks scary, you can do it manually:

  git log --oneline feature/deploy -- deploy/
  # note the commit hashes
  git checkout -b feature/deploy-clean master
  git cherry-pick abc123 def456 ...

  Then nuke the messy branch and rename:

  git branch -D feature/deploy
  git branch -m feature/deploy-clean feature/deploy

● That's the exact use case reposurgeon was built for - Eric S. Raymond wrote it to consolidate ancient repos with mixed VCS history.

  Fair warning though: reposurgeon has its own DSL and the learning curve is steep. It's less "run a command" and more "write a surgery script." The manual is a book.

  Quick sanity check before you dive in:

  dotfiles (git, current)
  ├── awesomewm (git)
  ├── vim (git)
  ├── spacemacs (git)
  ├── neovim (git)
  ├── gvim-1 (git, converted from hg?)
  ├── gvim-2 (git)
  └── ancient-gvim (hg, 2008, windows)

  The hg→git conversion is the interesting part. Options:

  1. reposurgeon - full control, can interleave timelines, fix author mappings, graft histories
  2. hg-fast-export + git grafts - simpler, just converts then you graft manually
  3. hg-git - mercurial extension, quick and dirty

  If you just want linear history (each repo becomes a subdirectory with its history preserved), git filter-repo can do most of it after converting hg→git. Reposurgeon shines when you need to interleave commits by timestamp across repos.

  How tangled is the history you want? Linear per-directory, or true unified timeline?

> notify! exists because you can't use :notifies from :define 

● Ah, right - define is a compile-time macro, not a real resource. It doesn't have notifies because there's no resource instance to attach the notification to.

  So notify! is a workaround: wrap the notification in a local_ruby_block (which IS a resource) just to get access to notifies.

  That's worth documenting:

  # Can't do this in a define:
  define :foo do
    notifies :reload, "service[bar]"  # ERROR: no method 'notifies'
  end

  # So we do this instead:
  define :foo do
    notify! "reload@service[bar]"  # creates a local_ruby_block that can notifies
  end

  The 8-notifications issue then comes down to whether mitamae deduplicates those local_ruby_block resources by name, or creates 8 separate ones. If it's 8 separate blocks, each fires its own notification, and if delayed notifications aren't deduplicated either... you get 8 template renders.

  Might be worth checking mitamae source for how it handles duplicate resource names. Or just accept the tax and move on - 8 renders of the same template is wasteful but not broken.

● git clone only creates a local branch for the default branch (master). The others exist only as remote-tracking refs (origin/dist, origin/notes, etc.), not local branches.

  git push --prune --branches means: "delete remote branches that have no matching local branch."

  Since you only have master locally, it wants to nuke everything else.

  # What you have after clone:
  git branch          # master (local)
  git branch -r       # origin/master, origin/dist, origin/notes, origin/WIP/self-deploy (remote-tracking)

  # git fetch doesn't create local branches, just updates remote-tracking refs

  Safe options:

  1. Don't use --prune (or be very careful with it)
  2. Create local tracking branches for everything you want to keep:
  git branch dist origin/dist
  git branch notes origin/notes
  # etc
  3. Use git push origin --prune :<branch> to explicitly delete specific branches only

  The foot-gun is that --prune compares against local branches, not remote-tracking refs. It's designed for "my local branches are the source of truth, delete everything else" workflows.

● Yeah, this is a fundamental design issue. Let me think through the options:

  The core problem: You want "collect data from many places, render once at the end, reload if changed."

  Current flow:
  cookbook A: push data → notify! → template queued
  cookbook B: push data → notify! → template queued (again)
  ...
  end of each cookbook: render template (wasteful)

  Ideal flow:
  cookbook A: push data
  cookbook B: push data
  ...
  end of run: render template once → reload if changed

  Option 1: Finalize recipe

  Remove all notify! calls. Just push data. Then have a final recipe that renders:

  # cookbooks/pf_finalize/default.rb (runs LAST in run_list)

    notifies :run, "execute[reload_pf]"

● The math:

  - Current: 24fps, ~6 seconds = ~144 frames
  - Each connection = 1 process doing usleep() + write()
  - 1000 connections = 1000 processes

  usleep() processes are basically free (sleeping), but 1000 forks still hurt.

  Defenses you probably already have:

  # sshd limits
  MaxStartups 10:30:60    # start refusing at 10, refuse 30% at 30, 100% at 60
  MaxSessions 10

  pf rate limiting:

  # In pf.conf
  pass in on $if_public proto tcp to port 22 \
      keep state (max-src-conn 10, max-src-conn-rate 5/10)

  10 connections per IP, max 5 new connections per 10 seconds.

  The real honeypot tuning:

  fps = 1      # 1 frame per second
  frames = 144 # 2+ minutes of dick
