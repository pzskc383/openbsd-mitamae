● You're hitting on one of the fundamental pain points of infrastructure-as-code: testing without production.

  How other people deal with it

  Chef/Puppet/Ansible ecosystem:
  - Test Kitchen + Vagrant/Docker for local VM testing
  - ServerSpec/InSpec for validation
  - CI/CD pipelines that spin up test VMs, converge, run tests, destroy

  Your challenge: Mitamae is much smaller/simpler than Chef. The testing ecosystem doesn't really exist for it.

  Common approaches in your situation:

  1. VM-based testing (what you're describing):
    - Local VMs via libvirt/Vagrant
    - Packer builds OpenBSD base images
    - Test Kitchen orchestrates: provision VM → run mitamae → validate → destroy
    - Need mock DNS/network infrastructure
  2. Immutable infrastructure (what many people do):
    - Build entirely new servers from scratch
    - Test on staging/dev servers that mirror production
    - Blue-green or canary deployments
    - Your "nuke and rebuild" approach is actually close to this
  3. Docker/containers (doesn't work for you):
    - Great for testing application configs
    - Terrible for testing system-level stuff (DNS, mail, kernel tunables, pf)
    - OpenBSD in Docker is... not really a thing
  4. Just YOLO it (surprisingly common):
    - Dry-run mode (hocho apply --dry-run)
    - Good backups
    - Configuration versioned in git
    - Accept that production is your test environment
